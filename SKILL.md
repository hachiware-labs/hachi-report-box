# Hachi Report Box

Use this skill to collect final report artifacts from one project into a central
`hachi-report-box` repository, keep a stable index, and optionally commit/push
the update so a personal static site can render it.

## Workflow

1. Identify the source artifacts to archive. Prefer explicit files or output
   directories from the current task. Do not copy build caches, dependency
   directories, or unrelated project files.
2. Locate the report box repository in this order:
   - A path explicitly provided by the user.
   - `HACHI_REPORT_BOX_DIR`, then `HACHI_REPORT_BOX_REPO`.
   - `~/workspace/hachi-report-box`.
   - Ask the user only if none of these exists.
3. Inspect `git status --short` in the report box repository before writing.
   Work with existing changes, and avoid overwriting unrelated user edits.
4. Use the bundled helper for normal collection:

   ```bash
   python scripts/hachi_report_box.py collect \
     --box-dir /path/to/hachi-report-box \
     --project my-project \
     --title "Short report title" \
     --kind report \
     /path/to/generated-report.md /path/to/output-directory
   ```

5. Add `--commit` when the user asked to commit. Add `--push` only when the
   user asked to push or when push is clearly part of the requested workflow.
6. Report the created entry path, index path, commit hash if committed, and
   push result if pushed.

## Storage Convention

The helper writes entries under:

```text
reports/<project-slug>/<yyyy>/<yyyy-mm-dd>/<entry-slug>/
```

Each entry contains:

- `files/`: copied source artifacts.
- `manifest.json`: metadata, source paths, copied files, sizes, and SHA-256
  hashes.

The helper regenerates:

- `reports/INDEX.md`: human-readable latest-first index.
- `reports/index.json`: static-site-friendly index data.

## Commit Discipline

When committing manually, stage only the new entry and regenerated indexes.
Do not stage unrelated files from the report box repository. Use concise
messages such as:

```text
Add report: <title>
```

If the report box repository has unrelated dirty files, leave them untouched
and mention them in the final response.

## Safety

Treat collected documents as personal but potentially publishable. Before
commit or push, quickly scan file names and obvious text content for secrets,
credentials, private keys, tokens, or unintended personal data. If sensitive
content is likely, stop and ask the user how to proceed.
